## Context

메인 랜딩은 founder-first 구조와 CTA 위계는 이미 정리되어 있다. 현재 문제는 기능 부족보다 카피 레벨의 불일치에 가깝다. Hero는 `믿고 맡길 수 있는 파트너`를 말하지만, About은 `같이 정리` 쪽으로 기울어 있고, 서비스 섹션은 일부가 패키지 소개형이며, 하단에는 내부 정책처럼 읽히는 trust-policy 섹션이 남아 있다.

이 change는 레이아웃 개편보다 카피와 섹션 역할을 정돈하는 작업이다. 메인 구매 의도는 `상담이 유용한가`보다 `이 사람에게 일을 맡겨도 되는가`에 가깝기 때문에, 신뢰 근거와 서비스 설명 모두 위임 가능성과 실행 범위를 더 직접적으로 보여줘야 한다.

## Goals / Non-Goals

**Goals:**
- Hero, About, Services, Contact를 같은 언어 레벨로 맞춘다.
- `trust-policy`처럼 내부 운영 기준을 방문자용 섹션으로 노출하지 않도록 정리한다.
- 서비스 카드 문장을 founder의 구매 판단 기준에 맞게 다듬는다.
- founder 전용 카피와 메인 홈 카피를 같은 톤으로 정렬한다.

**Non-Goals:**
- 서비스 구성 자체를 추가/삭제하지 않는다.
- CTA 구조나 라우팅을 다시 설계하지 않는다.
- 데이터 모델, 관리자 기능, 폼 구조는 변경하지 않는다.
- 외국인 개발자 페이지의 메시지 구조를 이번 change에서 재설계하지 않는다.

## Decisions

### 1. 카피 변경은 `의미 유지 + 위임 가능성 강화` 원칙으로 진행한다
- 이유: 이미 서비스 구조와 CTA 흐름은 정리되어 있다. 이번에는 구매 의도에 더 맞게 문장을 바꾸는 것이 목표다.
- 대안: Hero/서비스 구조를 다시 설계한다.
- 기각 이유: 구조 재설계는 범위가 커지고, 이번 문제의 핵심인 문장 질감과 신뢰 흐름을 바로 해결하지 못한다.

### 2. public trust는 `자연스러운 신뢰 단서`로만 보여준다
- 이유: 경력, 프로젝트 이력, LinkedIn 링크는 충분한 신뢰 근거다. 별도 정책 섹션은 방문자 가치를 거의 만들지 못한다.
- 대안: trust-policy 섹션을 유지하되 문장만 다듬는다.
- 기각 이유: 정책 섹션 자체가 사용자 관점과 맞지 않는다.

### 3. 서비스 카드는 `언제 맡기면 되는가`가 먼저 보이게 다듬는다
- 이유: founder는 서비스 이름보다 자기 상황 적합성을 먼저 본다.
- 대안: 카드 제목은 유지하고 summary만 부분 수정한다.
- 기각 이유: summary만 손보면 Hero/Contact와의 구매 흐름이 여전히 약할 수 있다.

### 4. founder route copy도 메인 홈과 같은 언어 레벨로 정리한다
- 이유: `/`와 founder 관련 하위 경로에서 다른 사람이 쓴 듯한 어조가 보이면 신뢰가 떨어진다.
- 대안: 메인 홈만 우선 수정한다.
- 기각 이유: 같은 founder journey 내 카피 일관성이 깨진다.

## Risks / Trade-offs

- [리스크] 너무 `맡길 수 있음`만 강조하면 단순 외주처럼 읽힐 수 있다.
  - 완화: Hero 아래와 About에서 사업 이해와 범위 조율 맥락을 같이 유지한다.
- [리스크] trust-policy를 제거하면 내부 증빙 기준이 사라진 것처럼 느껴질 수 있다.
  - 완화: 관련 기준은 spec과 source data에 남기고, public page에는 자연스러운 신뢰 단서만 노출한다.
- [리스크] 서비스 카피를 너무 짧게 줄이면 scope boundary가 흐려질 수 있다.
  - 완화: 카드 본문은 짧게 유지하되 기간/비용/권장 고객/인수인계 같은 bounded info는 유지한다.

## Migration Plan

1. draft 문서에서 섹션별 카피를 먼저 확정한다.
2. `landing/content.py`와 `landing/templates/landing/index.html`에 카피 및 섹션 정리를 반영한다.
3. founder 관련 테스트 문자열을 업데이트한다.
4. `./scripts/verify.sh`로 검증한다.

## Open Questions

- founder 전용 별도 페이지(`/for-founders/`)가 메인 홈과 다른 깊이의 메시지를 가져야 하는지는 구현 전에 한 번 더 확인이 필요하다.
- 서비스 카드에 `이럴 때 적합` 한 줄을 명시적으로 추가할지, 기존 summary만 조정할지는 구현 시 화면 밀도를 보고 결정한다.
