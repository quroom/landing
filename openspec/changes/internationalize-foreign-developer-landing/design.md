## Context

현재 랜딩은 페르소나 라우팅(`/`, `/for-foreign-developers/`)은 분리되어 있으나 템플릿과 콘텐츠 구조가 단일 언어(한국어) 중심이다. 이번 변경은 외국인 개발자 전용 페이지를 영어 기본 언어로 전환하고, 이후 랜딩 전체 다국어 운영이 가능하도록 i18n 구조를 정립하는 것이 목적이다.

제약 조건은 다음과 같다.
- 기존 URL 구조와 전환 흐름을 유지해야 한다.
- 기존 capability(페르소나 분리, foreign developer practical-linkage)의 의미를 깨지 않고 확장해야 한다.
- 템플릿/콘텐츠/폼 텍스트/테스트까지 동일한 언어 정책이 적용되어야 한다.

## Goals / Non-Goals

**Goals:**
- `/for-foreign-developers/` 문서 기본 언어를 영어로 정의하고 일관되게 노출한다.
- 랜딩 페이지 전반에 다국어 선택, 기본값, 폴백 규칙을 명시 가능한 i18n 계약을 수립한다.
- 번역 누락 시 동작(폴백/로그)을 예측 가능하게 만든다.
- 기존 페르소나별 CTA/FAQ/KPI 구조를 언어별로 일관되게 유지한다.

**Non-Goals:**
- 외부 번역 SaaS 도입 또는 자동 번역 파이프라인 구축
- 관리자 CMS 전면 개편
- 비랜딩 영역(관리자 페이지/이메일 템플릿 전체)의 일괄 다국어화

## Decisions

### 1) Locale 해석 우선순위를 명시한다
- 결정: `lang` 쿼리 파라미터 -> 저장된 사용자 선택(세션/쿠키) -> 페이지 기본 언어 순으로 locale을 결정한다.
- 이유: 캠페인 링크에서 강제 언어 지정이 가능하고, 재방문 시 사용자 선택을 존중할 수 있다.
- 대안: 브라우저 `Accept-Language` 우선.
- 대안 미채택 이유: 운영자가 의도한 랜딩 언어 실험 및 CTA 추적 일관성이 약해질 수 있다.

### 2) 페이지 기본 언어를 capability로 고정한다
- 결정: `/`는 한국어 기본, `/for-foreign-developers/`는 영어 기본으로 스펙에 고정한다.
- 이유: 페르소나 의도와 유입 트래픽의 기대 언어가 다르며, foreign developer 페이지는 영어-first가 전환에 유리하다.
- 대안: 사이트 전체 단일 기본 언어.
- 대안 미채택 이유: 페르소나별 정보 전달 효율이 떨어지고 카피 전략과 충돌한다.

### 3) 콘텐츠는 키 기반 언어 사전을 사용한다
- 결정: 텍스트 자산(헤더, CTA, FAQ, 폼 라벨/플레이스홀더)을 언어 키(`ko`, `en`) 기반 구조로 관리하고, 렌더 시 locale에 맞게 조회한다.
- 이유: 템플릿 분기 난립을 줄이고 테스트 가능성을 높인다.
- 대안: 템플릿별 하드코딩 분기.
- 대안 미채택 이유: 페이지 증가 시 유지보수 복잡도와 누락 위험이 커진다.

### 4) 폴백과 결측 처리 정책을 강제한다
- 결정: 요청 locale 번역이 없으면 페이지 기본 언어로 폴백하고, 기본 언어도 없으면 안전 기본(`ko`)으로 폴백한다. 결측 키는 로깅 가능한 이벤트로 남긴다.
- 이유: 사용자에게 빈 텍스트 노출을 방지하고 운영 추적성을 확보한다.
- 대안: 결측 시 원문 키 그대로 출력.
- 대안 미채택 이유: 사용자 경험 저하와 신뢰 하락 위험이 크다.

## Risks / Trade-offs

- [Risk] 텍스트 키 마이그레이션 중 누락 발생 가능
  -> Mitigation: 언어 키 커버리지 테스트와 결측 키 로깅을 추가한다.
- [Risk] 언어 파라미터와 기존 분석 이벤트의 차원 불일치
  -> Mitigation: 이벤트에 locale 속성을 포함하고 기존 대시보드 필터를 업데이트한다.
- [Risk] 초기 번역 품질 편차
  -> Mitigation: foreign developer 페이지 핵심 CTA/FAQ는 수동 리뷰 체크리스트를 운영한다.

## Migration Plan

1. 스펙 확정 후 locale 해석 유틸과 콘텐츠 구조를 도입한다.
2. `/for-foreign-developers/`부터 영어 기본 적용 및 회귀 테스트를 먼저 수행한다.
3. `/` 및 공통 폼/네비 텍스트를 i18n 구조로 확장한다.
4. 결측 키 로깅과 locale 이벤트를 점검한 뒤 배포한다.
5. 이상 발생 시 locale 선택 로직을 비활성화하고 페이지 기본 언어만 사용하는 모드로 롤백한다.
