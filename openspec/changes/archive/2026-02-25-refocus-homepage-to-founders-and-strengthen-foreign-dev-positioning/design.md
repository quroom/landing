## Context

현재 랜딩은 메인/창업자/외국인 개발자 페이지가 존재하지만, 메시지 우선순위와 증거(신뢰) 블록이 충분히 분리되지 않아 메인 페이지의 타겟이 혼재되어 보인다. 이번 변경은 정보 구조를 재배치하는 콘텐츠 중심 변경이며, 기존 Django 구조(`landing/content.py`, `landing/views.py`, `landing/templates/landing/*.html`)를 유지한 채 페르소나별 카피/CTA/신뢰 요소를 명확히 분리하는 것이 핵심이다.

핵심 이해관계자:
- 창업자/소상공인: 빠른 서비스화/외주 실행 + AX 도입 상담 필요
- 외국인 개발자: 한국 실무 연계/취업 연계에 대한 신뢰 근거 필요
- 운영자(대표): 실제 경력/네트워크/자격 기반으로 신뢰를 전달하되 과장 표현은 지양

기준 이력 타임라인(문구 고정 소스):
- 2011.01~2012.07: 삼성 소프트웨어 멤버십
- 2012.08~2014.10: 삼성전자
- 2020.03~현재: 큐룸

## Goals / Non-Goals

**Goals:**
- 메인 페이지를 창업자 전용 가치 제안 중심으로 재정렬한다.
- 메인/창업자 페이지에 AX(OpenClaw/바이브코딩) 기반 실행 서비스 오퍼를 명확히 배치한다.
- AX 서비스 오퍼를 패키지 단위로 고정한다(AX 진단 90분, 자동화 구축 2주, 실행 코칭 4주).
- 삼성전자 근무 경력, 전남대 출신, 공인중개사, 개발사 네트워크 등 신뢰 요소를 페이지 목적에 맞게 배치한다.
- 외국인 개발자 전용 페이지에 실무 연계/취업 연계 동선과 대기업 실무 경험 + 지역/네트워크 기반 지원 강점을 구조적으로 노출한다.
- 페르소나별 CTA와 FAQ가 서로 섞이지 않도록 분리 규칙을 강제한다.
- 페르소나별 KPI를 분리 측정한다(창업자: 상담 전환, 외국인: 연계 문의/면담 전환).
- OpenSpec 랜딩 스펙 문서(`quroom-landing-spec.md`)와 실제 페이지 콘텐츠를 동기화한다.

**Non-Goals:**
- 백엔드 아키텍처 변경 또는 신규 앱 도입
- 문의 폼 처리 로직 변경(전송 방식/저장 방식 유지)
- 디자인 시스템 전면 개편(Tailwind/기존 레이아웃 프레임 유지)
- 다국어 전체 리라이팅

## Decisions

### Decision 1: Homepage = founder-first single narrative
- 메인 페이지의 핵심 카피/서비스 카드/CTA를 창업자 전환 목표에 맞춰 구성한다.
- 메인 페이지에는 외국인 개발자 실무 연계 상세를 배치하지 않고, 외국인 전용 페이지로만 안내한다.
- 메인 페이지 서비스 섹션은 AX 실행 오퍼(아이디어 검증 자동화, 업무 자동화, MVP 가속) 중심으로 구성한다.
- 이유: 메인의 목적을 단일화하면 전환 경로가 명확해지고 메시지 충돌이 줄어든다.
- 대안: 메인에서 두 페르소나를 동등 노출하는 방식은 정보량이 늘어 집중도가 떨어져 제외.

### Decision 2: Credibility signals are page-scoped, not globally duplicated
- 공통 신뢰 요소(삼성전자 경력)는 메인 About/신뢰 섹션에 1회 강조한다.
- 외국인 전용 페이지에는 전남대 출신, 대기업 실무 경험, 개발사 네트워크, 공인중개사 기반 생활/현장 이해를 실무 연계 컨텍스트와 함께 배치한다.
- 신뢰 문구는 검증 가능한 사실만 사용하고, 추정/예정 표현은 분리 표기한다.
- 이유: 같은 이력을 모든 페이지에 반복하면 설득력이 약해지고 페이지 목적이 흐려진다.
- 대안: 모든 신뢰요소를 공통 블록으로 묶는 방식은 타겟별 relevance가 낮아 제외.

### Decision 3: Data-driven copy structure in `landing/content.py`
- 뷰/템플릿 하드코딩보다 `landing/content.py`의 페르소나 데이터 구조를 확장해 메시지를 관리한다.
- `founder`와 `foreign_developers` 데이터 필드를 분리해, AX 서비스 오퍼와 외국인 실무 연계 카피가 교차 노출되지 않도록 한다.
- 템플릿은 구조를 소비하는 형태로 유지하여 이후 문구 수정 비용을 최소화한다.
- 이유: 운영 중 문구 실험(A/B 성격의 문구 조정)과 OpenSpec 문서 싱크가 쉬워진다.
- 대안: 템플릿 직접 수정 중심은 단기 빠르지만 장기 유지보수 비용이 커 제외.

### Decision 4: Spec-first consistency enforcement
- `quroom-landing-spec.md`에 페르소나별 메시지/신뢰요소/CTA 분리를 명시하고 구현 시 동일 필드를 사용한다.
- 기존 main specs(`persona-landing-pages`, `landing-spec-persona-governance`, `service-content-completeness`)는 delta spec으로 강화한다.
- 이유: 문서와 구현 간 drift를 방지하고 이후 변경 시 검증 기준을 유지한다.

### Decision 5: Scope boundary and evidence governance
- 외국인 전용 페이지의 지원 범위를 명확히 구분한다: 실무/취업 연계 지원은 포함, 비자/법률 대행은 범위 외.
- 개발사 네트워크는 실명 대신 범주형 표현(예: 다수 개발사 협업 네트워크) 기본 정책을 사용한다.
- 경력/프로젝트/강점 항목에는 가능한 증빙 링크 타입(LinkedIn/포트폴리오/GitHub)을 지정한다.
- 이유: 신뢰성과 리스크를 동시에 관리하면서 운영 가능한 문구 정책이 필요하다.

## Risks / Trade-offs

- [Risk] 경력/자격 문구가 과장 또는 오해 소지가 있을 수 있음 -> Mitigation: 사실 기반/검증 가능한 표현만 사용하고 정량 문구는 보수적으로 유지
- [Risk] 메인 페이지에서 외국인 타겟 노출이 줄어 유입 손실 가능 -> Mitigation: 상단 네비/중간 브릿지 CTA로 전용 페이지 유입 유지
- [Risk] 콘텐츠 변경 범위가 넓어 템플릿/콘텐츠 불일치 가능 -> Mitigation: `content.py` 중심으로 데이터 정의 후 템플릿 참조 키 점검
- [Risk] 범위 오해로 비자/법률 요청이 유입될 수 있음 -> Mitigation: 외국인 전용 페이지 FAQ/문의 전 안내 문구에 범위 외 항목 명시
- [Risk] 네트워크/경력 신뢰성 검증 요청 대응 부족 -> Mitigation: 증빙 링크 정책과 공개 가능한 레퍼런스 목록 유지
- [Trade-off] 타겟 집중도는 높아지지만 단일 페이지에서 전달하는 범용성은 일부 감소

## Migration Plan

1. `quroom-landing-spec.md`에 메인(창업자+AX) / 외국인(실무 연계 중심) 역할 분리 원칙 반영
2. 경력 타임라인(삼성 소프트웨어 멤버십/삼성전자/큐룸)을 표준 문구로 정의하고 배치 위치를 고정
3. `landing/content.py`의 공통/페르소나 콘텐츠 모델에 AX 오퍼 필드와 외국인 실무 연계 필드를 분리 정의
4. 메인 AX 패키지(AX 진단 90분, 자동화 구축 2주, 실행 코칭 4주)를 서비스 카드로 반영
5. `landing/templates/landing/index.html`을 founder-first + AX 서비스 CTA 중심으로 정렬
6. `landing/templates/landing/foreign_developers.html`에 전용 강점 블록(실무 연계/경력/네트워크/정착 이해)과 범위 외 문구(비자/법률 대행 제외) 추가
7. 페르소나별 FAQ 분리(창업자: 범위/비용/일정, 외국인: 연계 프로세스/준비사항/지원 범위)
8. 페르소나별 KPI 측정 항목과 이벤트 이름 정의
9. 로컬 검증: 주요 경로 렌더링, CTA 링크, 신뢰 문구/증빙 링크 노출 위치 점검
10. 필요시 문구 조정 후 tasks로 구현 단위 확정
