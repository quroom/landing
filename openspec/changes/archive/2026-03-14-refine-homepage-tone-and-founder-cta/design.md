## Context

메인 랜딩(`/`)은 이미 구조상 창업자 우선 페이지지만, 실제 카피와 CTA 위계는 아직 창업자 관점으로 충분히 정리되지 않았다. 특히 Hero와 서비스 소개 구간에서 `OpenClaw`, `바이브코딩`, `자동화 실행` 같은 내부 실행 방식이 먼저 노출되면서, 창업자가 즉시 이해하는 문제 해결 언어와 결과 언어가 약해진 상태다.

이 change의 핵심 이해관계자는 두 그룹이다. 첫째는 실제 매출 전환 대상인 창업자/소상공인이고, 둘째는 페이지를 운영하며 후속 상담과 외주 범위를 조정해야 하는 운영자다. 따라서 문구는 창업자에게는 빠르게 이해되어야 하고, 운영자에게는 범위와 기대치를 명확히 정렬시켜야 한다.

이번 변경은 데이터 모델이나 라우팅 구조를 바꾸는 작업이 아니다. `landing/content.py` 기반의 카피 소스, `landing/templates/landing/index.html`의 CTA 배치, 그리고 메인/외국인 페이지 간 역할 분리 인상을 조정하는 콘텐츠 구조 변경이다.

## Goals / Non-Goals

**Goals:**
- 메인 홈페이지를 창업자 매출 전환에 유리한 톤으로 재정렬한다.
- 도구/방법론 중심 문구를 창업자 문제/결과 중심 문구로 치환한다.
- Hero와 Contact에서 첫 행동을 `상담 문의하기` 중심으로 분명히 한다.
- 무료 진단이나 서비스 상세 보기 같은 보조 CTA는 유지하되 시각적 우선순위를 낮춘다.
- 외국인 개발자 페이지는 별도 페르소나 페이지로 유지하면서 메인 페이지의 창업자 우선성을 흐리지 않게 한다.

**Non-Goals:**
- 서비스 가격, 일정, 제공 범위를 새로 설계하지 않는다.
- 외국인 개발자 페이지의 staged CTA 구조를 다시 설계하지 않는다.
- 새로운 페이지, 라우트, 폼, 데이터 저장 구조를 추가하지 않는다.
- 실제 카피 최종안 전체를 design 문서에서 확정하지 않는다. 카피 톤과 CTA 위계 기준만 정의한다.

## Decisions

### 1. 메인 카피는 문제/성과 언어를 우선하고 내부 실행 방식은 후순위로 내린다

`OpenClaw`, `바이브코딩`은 차별점이 될 수 있지만, 창업자 입장에서 첫 판단 기준은 아니다. 첫 화면에서 중요한 것은 "무엇이 해결되는가", "어떤 속도로 실행되는가", "누가 맡기 좋은가"다. 따라서 Hero, About, 서비스 카드의 요약 문구는 창업자의 실행 병목과 사업 결과를 먼저 말하고, 내부 도구나 방법은 필요 시 후속 설명 또는 세부 서비스 문구에서만 보조적으로 다룬다.

대안으로 기존 표현을 유지하고 부연 설명만 추가하는 방법도 있었지만, 이는 핵심 문구의 방향을 바꾸지 못해 첫인상 문제를 그대로 남긴다. 이번 change에서는 내부 방식 설명을 전면이 아니라 후면으로 보내는 쪽을 선택한다.

### 2. Hero CTA는 상담 중심 1순위, 진단/서비스 조회는 보조 액션으로 재배치한다

창업자 페이지의 전환 목표는 상담 연결이다. 무료 진단이나 서비스 보기 링크가 동일한 시각적 무게를 가지면 초기 의사결정이 분산된다. 따라서 Hero에서는 `상담 문의하기`를 유일한 주 CTA로 유지하고, `무료 자동화 실행 진단`과 `서비스 보기`는 보조 버튼 또는 텍스트 링크 수준으로 낮춘다.

대안으로 무료 진단을 여전히 동급 버튼으로 두는 방법도 있었지만, 현재 매출 우선순위와 상담형 프로젝트 전환 목적에는 맞지 않는다. 다만 상단 퍼널 유입을 완전히 버릴 필요는 없으므로 보조 행동으로는 남긴다.

### 3. Contact 구간은 설명보다 자격 정렬과 단일 다음 행동을 우선한다

메인 Contact 구간은 CTA를 설명하는 카드가 많을수록 다시 선택 피로를 만든다. 따라서 Contact 구간은 "누가 지금 문의하면 좋은가"를 다시 정렬하고, 기본 행동은 상담 문의로 모은다. 고액 외주/집중 트랙 구분은 폼 옵션이나 설명 텍스트 수준에서 안내한다.

대안으로 CTA를 여러 개 유지하되 설명을 더 붙이는 방법도 가능하지만, 랜딩 후반에서는 정보 추가보다 판단 단순화가 더 중요하다.

### 4. 메인 페이지와 외국인 페이지는 서로 다른 페르소나 목적을 유지한다

외국인 개발자 페이지는 계속 운영하되, 메인 페이지에서는 그 상세 내용을 노출하지 않는다. 메인 페이지는 창업자 중심 신뢰 허브로 유지하고, 외국인 개발자 페이지로의 이동은 보조 경로로만 둔다. 이렇게 해야 메인 페이지의 톤이 혼합되지 않고, 외국인 페이지도 별도 전환 퍼널로 유지된다.

대안으로 메인 페이지에서 두 페르소나를 동등하게 보이게 하는 방법도 있지만, 이는 현재 사용자 우선순위와 수익 구조 목표에 맞지 않는다.

## Risks / Trade-offs

- [도구 차별점이 너무 뒤로 밀릴 위험] → 서비스 카드 하단이나 상세 요약에서 실행 방식의 강점을 보조 근거로 유지한다.
- [무료 진단 CTA 약화로 상단 유입이 줄 수 있음] → Hero 보조 CTA나 Contact 하단 링크로 유지해 완전 제거는 피한다.
- [창업자 메시지가 너무 일반적이 될 위험] → 실제 작업 범위, 기간, 산출물 표현은 서비스 카드에서 그대로 유지해 구체성을 보완한다.
- [외국인 페이지 존재감이 줄어드는 위험] → 메인 페이지에서는 약하게 두되, 전용 페이지 안에서는 독립 CTA와 구조를 유지한다.

## Migration Plan

1. `landing/content.py`에서 메인 Hero, About, 서비스 카드, CTA 관련 카피를 founder-first 기준으로 수정한다.
2. `landing/templates/landing/index.html`에서 Hero와 Contact CTA 위계를 조정한다.
3. 외국인 개발자 진입 링크가 메인 페이지 보조 경로로 남는지 확인하고 필요 시 표현만 미세 조정한다.
4. 관련 테스트를 카피/CTA 구조 변경 기준에 맞게 갱신한다.
5. `./scripts/verify.sh`로 템플릿, 번역, 테스트 정합성을 확인한다.

## Open Questions

- `무료 자동화 실행 진단`은 Hero의 보조 버튼으로 남길지, Hero 밖의 섹션 링크로 더 낮출지는 구현 과정에서 실제 화면 밸런스를 보고 최종 확정한다.
