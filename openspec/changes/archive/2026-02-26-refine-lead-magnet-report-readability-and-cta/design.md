## Context

현재 진단 결과는 정보가 과다해 핵심 행동이 분산되고, 메일/화면 모두 줄간격과 항목 간 여백이 부족해 읽기 피로가 크다. 또한 preview 시나리오와 실제 결과 계산/표시 규칙이 완전히 동일하게 유지되지 않으면 운영자가 QA할 때 오해가 생길 수 있다.

이번 변경은 기능 확장보다 메시지 전달 품질과 전환 유도에 집중한다. 결과 자체를 `짧고 행동 중심`으로 재구성하고, 출력 일관성을 확보하는 것이 핵심이다.

## Goals / Non-Goals

**Goals:**
- 우선순위 제안을 Top 3으로 축소해 결정 피로를 줄인다.
- 화면/메일의 개행/간격을 개선해 가독성을 높인다.
- preview와 실제 진단 결과의 계산 및 포맷을 동일 규칙으로 맞춘다.
- 리포트 문구를 CTA 중심 구조로 변경해 상담 전환 유도력을 높인다.

**Non-Goals:**
- 진단 질문 체계(축/문항) 자체 변경
- 관리자 대시보드 기능 추가
- 외부 마케팅 도구/트래킹 신규 연동

## Decisions

### Decision 1: 우선순위 산출은 기존 로직을 유지하되 노출 개수만 3개로 제한
- 선택: 계산 로직은 안정성을 위해 유지하고, 출력 단계에서 Top 3만 표시
- 이유: 회귀 위험을 최소화하면서 사용자 체감 개선 효과를 빠르게 얻기 위해
- 대안:
  - 계산 로직 자체 재설계: 정확도 향상 여지는 있으나 범위가 커져 이번 목표에서 이탈

### Decision 2: 텍스트/HTML 리포트를 공통 블록 구조로 통일
- 선택: 요약 -> 우선순위 3 -> 즉시 실행 -> CTA 순으로 고정
- 이유: 채널(웹/메일)별 메시지 일관성을 확보하고 QA를 단순화하기 위해
- 대안:
  - 채널별 독립 포맷: 유연하지만 운영/테스트 비용 증가

### Decision 3: preview는 실제 생성 함수 재사용
- 선택: preview에서 별도 출력 문자열을 만들지 않고 실제 리포트 생성 함수를 직접 사용
- 이유: “preview는 맞는데 실전이 다름” 문제를 구조적으로 방지
- 대안:
  - preview 전용 템플릿/로직 유지: 표현 자유도는 높으나 불일치 위험 증가

### Decision 4: 메일 가독성은 문장 축약 + 시각 간격 동시 적용
- 선택: 문단 길이를 줄이고 HTML line-height/블록 margin을 확대
- 이유: 모바일 메일 클라이언트에서 읽기 효율 개선
- 대안:
  - 디자인만 개선: 문장 자체가 길면 효과 제한

### Decision 5: CTA 버튼은 1개만 제공하고 등급별로 분기
- 선택: 화면/메일 모두 단일 CTA 버튼만 노출하고, A/B/C 등급에 따라 버튼 문구/링크/보조카피를 분기
- 이유: 선택지를 줄여 클릭 결정을 단순화하고 등급별 행동 유도를 명확히 하기 위해
- 대안:
  - 복수 CTA 유지: 유연하지만 클릭 분산과 전환률 저하 가능성 증가

## Risks / Trade-offs

- [Risk] 정보 축약으로 일부 사용자가 “내용이 부족하다”고 느낄 수 있음
  - Mitigation: 메일/화면에 “상세 상담에서 확장 제공” CTA를 명확히 포함
- [Risk] Top 3 제한으로 놓치는 우선순위가 생길 수 있음
  - Mitigation: 내부 저장 데이터는 유지하고 화면만 Top 3 노출
- [Risk] 메일 클라이언트별 렌더 차이
  - Mitigation: 복잡 CSS를 피하고 inline 스타일 중심으로 간격 조정

## Migration Plan

1. 결과 생성 함수에서 우선순위 리스트를 Top 3로 제한
2. 결과 카드 템플릿 개행/여백 조정
3. 메일 텍스트/HTML 템플릿을 CTA 중심 구조로 재작성
4. preview 페이지가 동일 생성 함수 호출하는지 검증
5. 테스트 갱신 후 `./scripts/verify.sh` 실행
6. 문제 발생 시 메시지 포맷 변경만 롤백(계산 로직은 유지)
