## Context

현재 메인 랜딩 서비스 영역은 AX 패키지, 외주 집중 트랙, 부가 옵션이 같은 수준으로 섞여 있다. 이로 인해 방문자가 서비스 선택 경로를 빠르게 이해하기 어렵고, 창업 초기 고객이 실제로 필요한 "기본 인프라"(도메인/메일/홈페이지/문의 채널) 제안을 놓치기 쉽다.

현재 구현은 `landing/content.py`의 데이터 구조와 `landing/templates/landing/index.html`의 섹션 렌더링에 의존한다. 따라서 정보구조 변경은 데이터 모델(콘텐츠 키)과 템플릿 레이아웃을 함께 정렬해야 한다.

## Goals / Non-Goals

**Goals:**
- 메인 서비스 섹션을 `AX 실행 파트`, `외주용역 집중 트랙`, `창업 기본 인프라 구축`으로 분리한다.
- `창업 기본 인프라 구축` 오퍼를 창업 초기 필수 항목 중심으로 명확히 제시한다.
- 다음 스마트워크 기반 비즈니스 메일 오퍼를 `도메인 구매 / 연결 / DNS 레코드 설정` 범위로 명시한다.
- 서비스 카드 필수 표기(기간, 가격, 대상, 결과물/비고)의 일관성을 유지한다.

**Non-Goals:**
- 백엔드 API/DB 스키마 추가 또는 변경
- 외국인 개발자 전용 페이지의 구조 개편
- 결제/견적 자동화 기능 추가

## Decisions

### Decision 1: 서비스 구조는 콘텐츠 데이터 기반으로 분리 렌더링한다
- `content.py`에서 서비스 ID/그룹 키를 기준으로 분기하고, 템플릿은 그룹별 섹션을 명시적으로 렌더링한다.
- 이유: 이미 데이터 중심 렌더링 구조가 있어 변경 범위를 최소화할 수 있고, 향후 오퍼 추가 시 코드 영향이 작다.
- 대안: 템플릿에서 텍스트 매칭으로 임시 분기. 기각 이유: 유지보수 시 오타/카피 변경에 취약하다.

### Decision 2: 창업 기본 인프라는 옵션이 아니라 독립 파트로 취급한다
- `추가 구축 옵션` 표현 대신 `창업 기본 인프라 구축` 파트로 명시한다.
- 이유: 단발성 부가 기능이 아니라 창업 초기에 필수로 필요한 기반이라는 메시지를 전달하기 위함.
- 대안: 기존 옵션 박스 유지. 기각 이유: 중요도 전달이 약해 전환에 불리하다.

### Decision 3: 비즈니스 메일 오퍼는 다음 스마트워크 기준으로 고정한다
- 오퍼 문구를 `도메인 구매 / 연결 / DNS 레코드 설정`으로 단순·명확화한다.
- 이유: 국내 대상 이해도가 높고, 제공 범위 경계가 명확하다.
- 대안: Google Workspace/Microsoft 365 병렬 표기. 기각 이유: 초기 상담 단계에서 선택 복잡도 증가.

### Decision 4: 외주 집중 트랙은 AX 카드와 동일 레이아웃을 사용한다
- 카드 UI 패턴(요약, 범위 리스트, 기간/가격/대상, CTA)을 AX 파트와 동일하게 유지한다.
- 이유: 비교 가능성과 시각적 일관성이 높아진다.

## Risks / Trade-offs

- [Risk] 파트 분리 후 서비스 수가 늘어나 스크롤이 길어진다 → 상단 요약 문구와 섹션 제목을 명확히 배치해 탐색 비용을 줄인다.
- [Risk] 가격 표기가 상담에서 경직된 기준으로 인식될 수 있다 → 범위형 표기와 "협의" 문구를 유지한다.
- [Risk] 창업 기본 인프라 범위가 과도하게 확장될 수 있다 → 카드 비고에 포함/제외 범위를 명시한다.

## Migration Plan

1. `content.py`의 서비스/옵션 데이터 구조를 정렬한다.
2. `index.html` 서비스 섹션을 3파트 구조로 리렌더링한다.
3. 문구/가격/정책 텍스트를 현재 제안 기준으로 통일한다.
4. 주요 앵커(`#services`, `#contact`) 동작 및 모바일 레이아웃을 수동 검증한다.

Rollback:
- 템플릿과 콘텐츠 파일을 이전 커밋으로 되돌리면 즉시 복구 가능하다.
