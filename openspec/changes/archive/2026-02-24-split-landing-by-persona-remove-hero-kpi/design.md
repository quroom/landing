## Context

현재 Django 단일 랜딩(`templates/landing/index.html`)에 두 페르소나(창업자, 외국인 개발자) 메시지가 함께 존재해 CTA 집중도가 낮다. 콘텐츠 소스(`landing/content.py`)에도 공통/페르소나 데이터가 섞여 있으며, Hero의 `hero_kpi_text`는 내부 운영 지표 성격이라 외부 랜딩 신뢰 메시지와 목적이 다르다.

이 변경은 기존 스택(Django 템플릿 + 정적 자산 + 단순 라우팅)을 유지한 상태에서 콘텐츠 구조와 페이지 구조만 분리해 리스크를 낮추는 방식으로 진행한다.

## Goals / Non-Goals

**Goals:**
- 메인 랜딩에서 `hero_kpi_text` 노출을 제거한다.
- 창업자용 랜딩 페이지와 외국인 개발자용 랜딩 페이지를 분리 제공한다.
- 공통 회사 신뢰 정보는 메인 랜딩에 유지하고, 페르소나별로 메시지/CTA를 분리한다.
- 제공 서비스 섹션의 누락 항목을 식별하고 메인/페르소나 페이지에 맞게 보완한다.
- 기존 URL 동작(홈 진입)과 기본 SEO 메타 동작을 깨지 않고 확장한다.
- OpenSpec 문서(`quroom-landing-spec.md`)에 페이지 역할 분리 기준을 반영한다.

**Non-Goals:**
- 다국어 시스템 전체 도입(i18n 프레임워크 적용)
- 새로운 DB 모델 추가 및 관리자 CMS 구축
- 디자인 시스템 전면 개편(컬러/타이포 대규모 리브랜딩)
- 마케팅 자동화 도구(광고 픽셀, CRM) 연동

## Decisions

### Decision 1: URL 기반 페이지 분리
- 선택: `/`(메인), `/for-founders/`(창업자), `/for-foreign-developers/`(외국인 개발자) 3개 라우트 구성
- 이유: 가장 낮은 구현 복잡도로 명확한 진입 분리 가능, QR/캠페인 링크 운영이 쉬움
- 대안:
  - 쿼리 파라미터 방식(`/?persona=founder`): URL 공유/SEO/분석 분리가 약함
  - 단일 페이지 탭 전환: 초기 로딩은 단순하지만 CTA 분리와 측정이 불명확

### Decision 2: 콘텐츠 계층 분리(공통 + 페르소나)
- 선택: `landing/content.py`에서 공통 콘텐츠와 페르소나 콘텐츠 블록을 명시적으로 분리
- 이유: 템플릿 조건문 난립을 줄이고 유지보수성을 확보
- 대안:
  - 템플릿 하드코딩: 빠르지만 문구 수정 시 코드 변경 범위가 넓어짐
  - DB 기반 CMS: 유연하지만 현재 범위 대비 과도함

### Decision 3: Hero KPI 완전 제거(외부 노출)
- 선택: `hero_kpi_text`를 메인 렌더 경로에서 사용하지 않음
- 이유: 내부 KPI 문구는 외부 신뢰/가치 전달에 직접 기여하지 않고 메시지 노이즈를 유발
- 대안:
  - 문구를 약화해 유지: 여전히 목적 혼선 발생 가능
  - 숨김 토글: 운영 실수로 재노출 가능

### Decision 4: 기존 문의 폼 재사용 + CTA 분기만 추가
- 선택: 기존 문의 폼 파이프라인(폼 검증/메일 전송)은 유지, 페이지별 문의 유형 초기값 또는 CTA 문구만 분리
- 이유: 안정된 경로를 재사용해 회귀 리스크를 줄임
- 대안:
  - 페이지별 폼 완전 분리: 추적은 쉬우나 중복 코드 증가

### Decision 5: 서비스 콘텐츠 체크리스트를 명세와 코드에 동시 적용
- 선택: `quroom-landing-spec.md`의 서비스 정의를 기준으로 `landing/content.py`의 서비스 항목을 매핑하고, 누락 여부를 체크리스트로 검증
- 이유: 페이지 분리 과정에서 서비스 카드 누락/중복/카피 불일치를 방지
- 대안:
  - 구현 후 수동 확인: 빠르지만 누락 위험이 높음
  - 페이지별 별도 소스 관리: 독립성은 높지만 유지보수 비용 증가

## Risks / Trade-offs

- [리스크] 페이지 수 증가로 콘텐츠 불일치 발생 가능 → [완화] 공통 항목은 단일 소스(`content.py`)에서만 관리
- [리스크] 내부/외부 문서 불일치(`quroom-landing-spec.md` vs 실제 구현) → [완화] 이번 변경에서 문서와 코드를 함께 수정하고 체크리스트로 검증
- [리스크] SEO 중복(유사 문구 다수 페이지) → [완화] 페이지별 title/description을 구분하고 canonical 전략 적용 검토
- [리스크] 서비스 항목 누락/불일치로 제안 범위 오해 발생 → [완화] 서비스 카드별 필수 필드(서비스명/대상/범위/CTA)를 공통 검증
- [트레이드오프] 빠른 분리를 위해 DB/CMS 미도입 → 단기 민첩성은 높지만 비개발자 편집성은 낮음

## Migration Plan

1. `quroom-landing-spec.md` 업데이트: Hero KPI 제거, 메인/페르소나 페이지 역할 분리 규칙 + 서비스 누락 보완 기준 명시
2. `landing/content.py` 리팩터링: 공통/페르소나 콘텐츠 구조 분리, `hero_kpi_text` 제거, 서비스 항목 보완
3. `landing/views.py`/`landing/urls.py` 확장: 2개 페르소나 뷰/라우트 추가
4. `templates/landing/`에 페르소나 템플릿 추가, 기존 메인 템플릿에서 KPI 문구 제거
5. 로컬 검증: 각 URL 렌더/CTA/문의 폼 동작 + 서비스 카드 누락 여부 확인

Rollback:
- 문제 발생 시 신규 라우트와 템플릿 제거, 기존 단일 랜딩 경로로 즉시 복귀 가능

## Open Questions

- 외국인 개발자 페이지 1차 언어를 한국어 단일로 갈지, 영문 병행으로 시작할지 확정 필요
- CTA 도착 지점(이메일/LinkedIn/폼) 중 페르소나별 1순위 채널을 고정할지 여부
- 페이지별 GA4 이벤트 명명 규칙(예: `cta_founder_click`, `cta_foreign_click`)을 이번 변경 범위에 포함할지 여부
