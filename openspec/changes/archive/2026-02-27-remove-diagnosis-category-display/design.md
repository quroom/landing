## Context

무료 진단은 내부적으로 4개 축을 기반으로 점수/해석을 계산하지만, 사용자 화면과 결과 카피에서 카테고리 개념이 함께 노출되면서 문항 의도와 결과 메시지가 어긋나는 사례가 반복되고 있다.  
특히 같은 카테고리 내부 문항의 질문 결이 다르면, 사용자는 응답 체감과 결과 메시지 사이에 불일치를 느끼고 진단 신뢰도가 떨어진다.

## Goals / Non-Goals

**Goals:**
- 화면에서 카테고리 중심 노출을 제거하고 질문 중심 경험으로 통일한다.
- 질문 문구/의도 키/결과 카피를 같은 의미 체계로 맞춰 해석 불일치를 줄인다.
- preview/email/웹 결과의 공통 섹션 계약을 유지하면서 카테고리 의존 표현을 축소한다.

**Non-Goals:**
- 점수 모델 자체를 ML/통계 모델로 재설계
- 문의 접수 및 메일 발송 파이프라인 구조 변경
- 진단 문항 개수(8개) 및 필수/선택 구조 자체 변경

## Decisions

### 1) 카테고리는 내부 계산용으로만 유지하고 UI 노출은 제거한다
- 선택: `DIAGNOSIS_AXES`는 점수 계산/약점 축 판단에만 사용하고, 설문 렌더링은 질문 리스트만 노출한다.
- 이유: 사용자에게는 질문 의도만 보이게 하여 이해 경로를 단순화하고, 내부 계산 안정성은 유지한다.
- 대안: 카테고리를 유지하되 설명 문구만 축소.
  - 미채택 이유: 문항-결과 불일치 체감의 근본 원인을 해소하지 못한다.

### 2) 질문 의도(intent key)를 단일 진실원으로 사용한다
- 선택: 문항 메타(`intent_key`)를 기준으로 우선과제/추천툴/결과 문구를 매핑한다.
- 이유: 같은 입력이 항상 같은 해석 경로를 타도록 고정해 카피 드리프트를 줄인다.
- 대안: 카테고리 키와 질문 키를 혼합 사용.
  - 미채택 이유: 분기 증가로 메시지 일관성이 다시 깨질 가능성이 높다.

### 3) 카테고리 메시지는 축 대표 문항(anchor) 기반으로 생성한다
- 선택: 각 축에서 가장 우선도가 높은 문항을 anchor로 선택하고, 축 메시지 보조 문장에 anchor 문항 맥락을 포함한다.
- 이유: 축 단위 요약을 유지하면서도 응답 체감과 직접 연결되는 설명을 제공할 수 있다.
- 대안: 축 평균 점수만으로 고정 문구 출력.
  - 미채택 이유: 평균값만으로는 사용자가 “왜 이 메시지가 나왔는지”를 이해하기 어렵다.

### 4) 채널별 출력은 공통 섹션 AST 계약을 유지한다
- 선택: summary/weakest/one_action/tools/next_action 순서와 헤더를 고정하고, 웹·preview·이메일이 동일 AST를 사용한다.
- 이유: 메시지 동등성 테스트 지점을 단일화하고 회귀 검증을 단순화한다.
- 대안: 채널별 개별 템플릿 문구 유지.
  - 미채택 이유: 단기 수정은 쉬우나 장기적으로 불일치 회귀가 반복된다.

## Risks / Trade-offs

- [카테고리 비노출로 구조 인지가 어려울 수 있음] -> 결과 화면에서 약점 축 설명은 유지하되 질문/행동 중심 언어로 안내한다.
- [anchor 문항 선택 로직이 과도하게 단순해 보일 수 있음] -> 우선순위 계산식과 tie-break 규칙을 테스트로 고정한다.
- [기존 스냅샷 회귀 다수 발생 가능] -> 섹션 텍스트/이메일/preview 테스트를 함께 갱신하고 계약 테스트를 유지한다.

## Migration Plan

1. 질문 렌더링 경로에서 카테고리 그룹 출력 제거 및 질문 리스트 구조 고정
2. 질문 메타(intent/weight/order) 기준으로 결과 추천 로직 정렬
3. 축 메시지 생성 시 anchor 문항 연계 보조 문장 반영
4. 공통 섹션 AST 텍스트 계약(web/preview/email) 재검증 및 테스트 갱신
