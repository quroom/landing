## Context

리드마그넷 진단 결과는 최근 구조 단순화(요약 -> 핵심 보완 1개 -> 2주 과제 1개 -> CTA 1개)가 반영되어 정보량은 정리됐지만, 랜딩페이지 본문에서 약속한 가치 제안과 결과 카피의 전환 톤이 아직 완전히 정렬되지 않았다.  
특히 결과 화면/이메일/문의 구간이 채널별로 미세하게 다른 표현을 사용하면 신뢰가 분산되고, CTA 클릭 직후 맥락 단절로 문의 전환이 떨어질 수 있다.

## Goals / Non-Goals

**Goals:**
- 진단 결과 핵심 카피(요약/2주 과제/CTA)를 실행 중심 문구로 표준화한다.
- 결과 화면과 이메일이 동일 카피 계약(같은 섹션 모델)을 사용하도록 보장한다.
- CTA 클릭 이후 문의 구간에 진단 맥락이 이어지도록 기본 연결 상태를 강화한다.

**Non-Goals:**
- 점수 계산/등급 판정 로직 변경
- 진단 질문 구조/문항 변경
- A/B 실험 프레임워크 및 전환 측정 인프라 확장(트랙 B 범위)

## Decisions

### 1) 카피 표준은 공통 섹션 모델에서 관리한다
- 선택: 요약/과제/CTA 문구의 기준 문장을 공통 섹션 생성 경로에서 관리한다.
- 이유: 결과 화면과 이메일에서 같은 문구가 출력되어야 사용자 인지가 안정되고 QA 지점이 단일화된다.
- 대안: 채널별 템플릿에서 개별 문구 유지.
  - 미채택 이유: 문구 드리프트 회귀가 반복되고 리뷰 비용이 커진다.

### 2) 2주 과제는 "작업 1개 + 완료 기준" 포맷으로 고정한다
- 선택: 과제 라벨을 행동형으로 통일하고, 완료 기준(기간/검증 기준) 문장을 항상 포함한다.
- 이유: 추상 조언보다 실행 지시가 전환과 후속 상담 연결에 유리하다.
- 대안: 진단 문항 문장을 과제 타이틀로 그대로 노출.
  - 미채택 이유: 문장 종결형 서술이 과제 지시로 읽히지 않아 행동 유도가 약하다.

### 3) CTA는 단일 표현을 유지하되 문의 구간 맥락값을 전달한다
- 선택: CTA 문구와 note는 공통 표준(단일 표현)을 유지하고, 문의 진입 시 "진단 기반 문의" 컨텍스트가 `미리선택` 상태로 적용되도록 연결한다. 구현 복잡도가 예상보다 커지면 기존 문의 기본값을 유지한다.
- 이유: CTA 카피를 잦게 분기하면 메시지가 분산되므로, 맥락 전달은 라우팅/기본값에서 해결하는 것이 안정적이다.
- 대안: 등급별 CTA를 다중 분기해 강도를 조절.
  - 미채택 이유: 현재 트랙 A 범위 대비 복잡도가 커지고 측정 해석이 어려워진다.

### 4) 변경 검증은 카피 스냅샷 + 섹션 동등성으로 묶는다
- 선택: 테스트에서 섹션 순서/제목/CTA 링크 동등성과 핵심 카피 스냅샷을 동시에 검증한다.
- 이유: 구조 회귀와 문구 회귀를 함께 차단할 수 있다.
- 대안: UI 통합 테스트만으로 확인.
  - 미채택 이유: 실패 원인 분석이 느리고, 문구 단위 회귀를 놓치기 쉽다.

## Risks / Trade-offs

- [카피 강도가 과도해 일부 사용자에게 부담으로 인식될 수 있음] -> CTA note를 "지원 안내형" 톤으로 유지해 압박감을 낮춘다.
- [채널 동기화 중 문구 미세 차이 제거로 기존 반응 패턴이 변할 수 있음] -> 스냅샷 테스트로 변경 범위를 명확히 고정한다.
- [문의 맥락 기본값이 실제 문의 유형과 다를 수 있음] -> 사용자가 쉽게 변경 가능한 선택 UI를 유지한다.

## Migration Plan

1. 현재 결과 화면/이메일 카피를 공통 섹션 모델 기준으로 재정렬한다.
2. 2주 과제 문구를 "작업 1개 + 완료 기준" 규칙으로 통일한다.
3. CTA 이후 문의 진입 컨텍스트(기본값/문구 연결)를 점검하고 반영한다.
4. 테스트(`test_lead_magnet_sections`, `test_contact_form`, `test_landing_pages`) 기대값을 갱신한다.
5. 회귀 검증 후 적용하고, 필요 시 카피만 이전 버전으로 빠르게 롤백한다.
