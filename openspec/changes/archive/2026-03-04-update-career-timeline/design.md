## Context

홈페이지 신뢰 구간의 커리어 타임라인은 현재 개발 경력 일부만 보여주고 있어, 창업/운영/자동화 실행 경험의 연속성이 충분히 전달되지 않는다. 특히 2015~2017 구간(자격 취득, 중개업 운영, 자동화 도입, 쉐어하우스 창업/확장)은 현재 서비스 메시지와 직접 연결되는 근거인데 화면/데이터 구조에 반영되어 있지 않다.

이번 변경은 "경력 추가"보다 "스토리 일관성"을 목표로 한다. 즉 타임라인 항목 자체(연도/문구)와 표시 규칙(표현 톤, 순서, 검수 기준)을 함께 정의해 향후 문구 수정 시에도 메시지 드리프트를 막는다.

## Goals / Non-Goals

**Goals:**
- 2015/2016/2017 타임라인 항목을 canonical career 데이터에 반영한다.
- 2016 항목을 자동화 경험 중심 문구로 고정한다.
- 2017 항목을 `쉐어하우스 창업 및 확장`으로 정규화한다.
- 타임라인 카피를 `연도 + 행동 + 결과(또는 목적)` 톤으로 통일한다.

**Non-Goals:**
- 홈페이지 전체 레이아웃/디자인 개편.
- 포트폴리오 섹션 구조 변경.
- 커리어 기간 합산 로직의 계산 방식 자체 변경.

## Decisions

### 1) 타임라인 단일 소스는 `CAREER_RANGES`로 유지
- 선택: 기존과 동일하게 `landing/content.py`의 `CAREER_RANGES`를 canonical source로 사용한다.
- 이유: 템플릿/메트릭 계산(`career_duration`)이 이미 이 구조를 참조하고 있어 변경 파급이 작다.
- 대안: 별도 JSON/YAML 파일로 분리.
  - 미채택 이유: 현재 범위에서는 관리 포인트만 늘고 실익이 적다.

### 2) 2015~2017은 "연도 중심 라벨"로 명시
- 선택: 해당 항목은 월 단위보다 연도 메시지가 핵심이므로 라벨에 연도-행동 정보를 직접 포함한다.
- 이유: 방문자가 빠르게 스캔할 때 핵심 이력(자격, 자동화, 창업/확장)을 즉시 인지할 수 있다.
- 대안: 날짜 범위만 두고 설명을 별도 본문으로 분리.
  - 미채택 이유: 타임라인 자체의 전달력이 약해진다.

### 3) 2016 문구는 자동화 결과를 포함해 고정
- 선택: `2016: 중개업 활동, 매물 수집 자동화로 운영 시간 단축`을 기준 문구로 채택한다.
- 이유: 현재 서비스 가치(자동화 실행)의 출발 근거를 과장 없이 연결할 수 있다.
- 대안: 자동화 언급 없는 중립 문구.
  - 미채택 이유: 차별화 포인트가 사라진다.

### 4) 기존 경력 항목은 보존, 신규 항목만 삽입
- 선택: 기존 경력(삼성전자, 큐룸) 항목은 유지하고 2015~2017만 추가/정정한다.
- 이유: 이미 배포된 신뢰 지표와 충돌을 최소화한다.
- 대안: 타임라인 전면 재작성.
  - 미채택 이유: 검수 범위가 불필요하게 커진다.

## Risks / Trade-offs

- [Risk] 라벨이 길어져 모바일에서 줄바꿈이 늘어날 수 있음 → Mitigation: 템플릿에서 줄바꿈/간격 확인 및 필요 시 문구 길이 가이드 추가.
- [Risk] 연도 중심 라벨과 날짜 계산 로직 불일치 가능성 → Mitigation: 시작/종료값은 계산 목적에 맞춰 유지하고, 라벨만 스토리 전달 기준으로 관리.
- [Risk] 향후 문구 임의 변경으로 메시지 일관성 저하 → Mitigation: spec에 기준 문구와 표현 규칙을 명시.

## Migration Plan

1. `CAREER_RANGES`에 2015/2016/2017 항목을 추가하고 기존 항목과 순서를 정렬한다.
2. 템플릿 렌더 결과에서 타임라인 라벨과 `career_duration` 노출이 정상인지 확인한다.
3. 타임라인 관련 테스트(항목 존재/문구 포함)를 추가해 회귀를 방지한다.
4. 수동 검수: 홈페이지에서 타임라인 스캔 시 메시지 흐름(자격 취득 → 자동화 도입 → 창업/확장 → 현재 운영)이 자연스러운지 확인한다.
