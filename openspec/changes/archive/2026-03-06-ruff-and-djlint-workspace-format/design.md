## Context

현재 저장 시 포맷 결과가 에디터 기본 포맷터와 개발자별 확장 설정에 따라 달라집니다. 특히 Django 템플릿은 일반 HTML 포맷터가 템플릿 변수를 안전하게 다루지 못해 줄바꿈/개행 회귀가 발생할 수 있습니다. Python 쪽도 포맷/정렬 도구가 명시되어 있지 않아 import 순서와 코드 스타일이 PR마다 흔들리는 상태입니다.

이 change는 워크스페이스 단위에서 `ruff`(Python)와 `djlint`(Django template)를 표준 도구로 고정해, 로컬-리뷰-CI 결과를 일치시키는 것을 목표로 합니다.

## Goals / Non-Goals

**Goals:**
- Python 포맷/정렬/기본 lint 동선을 `ruff` 기반으로 통합한다.
- Django 템플릿 포맷을 `djlint`로 고정하고 HTML 기본 포맷터와 충돌을 제거한다.
- 워크스페이스 설정(예: VS Code)과 문서(README/운영 가이드)를 동일 기준으로 맞춘다.
- 검증 단계에서 포맷 회귀를 빠르게 감지할 수 있도록 스크립트 동선을 정리한다.

**Non-Goals:**
- 기존 코드 전체를 대규모 리포맷하는 작업은 이번 범위에서 제외한다.
- 프론트엔드 빌드 체인(Prettier/ESLint) 재설계는 이번 범위에 포함하지 않는다.
- IDE별(모든 에디터) 플러그인 가이드를 완전하게 제공하지 않는다.

## Decisions

- 선택: Python 표준 도구를 `ruff` 단일 도구로 설정한다.
- 이유: 포맷(`ruff format`)과 import 정렬/기본 lint(`ruff check --fix`)를 하나의 설정 파일로 관리할 수 있어 운영 복잡도가 낮다.
- 대안: `black + isort + flake8` 조합.
  - 미채택 이유: 설정 포인트와 실행 순서가 늘어나고 로컬 환경 차이 위험이 커진다.

- 선택: Django 템플릿(`landing/templates/**`)은 `djlint`를 기본 포맷터로 고정한다.
- 이유: 템플릿 변수/태그를 안전하게 다루며 HTML Language Features의 자동 개행 회귀를 줄일 수 있다.
- 대안: 템플릿에도 일반 HTML 포맷터 사용.
  - 미채택 이유: 템플릿 문법 깨짐 문제가 재발할 가능성이 높다.

- 선택: 검증은 “기존 verify(check/test) + 포맷 체크(ruff/djlint)”로 단계화한다.
- 이유: 포맷 회귀를 빠르게 감지하면서도 기존 테스트 흐름과 충돌하지 않는다.
- 대안: 포맷 적용을 수동 문서로만 안내.
  - 미채택 이유: 실행 누락으로 표준이 유지되지 않는다.

## Risks / Trade-offs

- [초기 도입 시 포맷 차이로 PR 변경량 증가] → 점진 도입(변경 파일 우선)과 1회 기준선 정리 전략 병행
- [개발자 IDE 설정 편차] → 워크스페이스 설정 파일과 README 명령 예시를 단일 소스로 유지
- [djlint 규칙 과잉 적용으로 템플릿 가독성 저하] → 필요한 규칙만 최소로 고정하고 예외 구간은 ignore 주석 정책 문서화

## Migration Plan

1. `ruff`/`djlint` 의존성과 설정 파일을 추가한다.
2. 워크스페이스 포맷터 매핑(파이썬=ruff, django-html=djlint)을 설정한다.
3. 검증 스크립트 또는 별도 포맷 스크립트에 체크 명령을 추가한다.
4. 문서에 로컬 실행 순서(`format`/`check`)와 템플릿 포맷 주의사항을 반영한다.
5. 샘플 파일 기준으로 실제 포맷 동작을 확인하고 회귀 테스트를 수행한다.

Rollback:
- 포맷 설정 충돌 시 워크스페이스 포맷터 매핑만 이전 상태로 되돌리고, 스크립트 변경은 유지해 수동 실행으로 임시 운용한다.

- CI에서 포맷 위반 시 auto-fix PR을 허용할지, 실패만 반환할지?
