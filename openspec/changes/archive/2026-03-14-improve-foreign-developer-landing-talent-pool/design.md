## Context

현재 외국인 개발자 랜딩은 단일 문의 폼 전환 구조다. 이 구조는 초기에 요구 정보가 많아 이탈이 발생하기 쉽고, 취업 준비 단계가 다양한 방문자(초기 탐색, 서류 준비 전, 적극 지원 중)를 동일 폼으로 처리해 후속 운영 분류가 어렵다.

이번 변경은 다음 제약을 가진다:
- 기존 Django 랜딩 구조(템플릿 + `content.py` + `ContactForm`)를 크게 벗어나지 않아야 한다.
- 기존 외국인 페이지의 신뢰/범위 경계(비자·법률 직접 대행 아님)를 유지해야 한다.
- 기존 분석 이벤트 및 운영 대시보드와 연계 가능한 데이터 흐름이어야 한다.

주요 이해관계자:
- 운영자(대표): 리드 선별, 후속 연락, 소개 진행 판단
- 외국인 개발자 사용자: 낮은 진입 장벽, 단계형 정보 제출
- 네트워크 파트너(개발사/지원 파트너): 프로필 핏이 맞는 후보만 전달받길 기대

## Goals / Non-Goals

**Goals:**
- 1차 저마찰 CTA와 2차 상세 프로필 CTA를 분리해 전환 퍼널을 단계화한다.
- 랜딩 카피를 취업 보장형이 아닌 전략/준비/연결 중심으로 재정렬한다.
- 광주/전남 로컬 우선 지원과 타지역 온라인 우선 지원 원칙을 명시해 운영 기대치를 정렬한다.
- 운영자가 후보 상태를 단계적으로 관리할 수 있도록 데이터 상태값을 정의한다.
- 커뮤니티 정보 공유 동선을 랜딩 내에서 명시해 장기 접점을 확보한다.
- 메인 창업가 Hero에서 주 행동을 창업자 상담으로 집중시킨다.

**Non-Goals:**
- 완전한 채용 플랫폼/잡보드 구축
- 자동 추천 매칭 엔진 도입
- 비자/법률 자문 직접 수행 기능
- 외부 CRM 전면 교체

## Decisions

### 1) 퍼널 분리: 단일 폼 -> 2단계 CTA
- 결정:
  - 1차 CTA는 최소 필드(닉네임/이메일/희망 직무)와 선택 메모(`추가로 남기고 싶은 말`)를 수집한다.
  - 2차 CTA는 라포 형성 이후 상세 매칭 정보(CV 또는 LinkedIn, GitHub 또는 포트폴리오, 주요 스택, 경력 수준, 비자 상태, 근무/지역 선호, 입사 가능 시점)를 수집한다.
- 이유:
  - 초기 마찰을 줄여 유입 대비 등록률을 높이고, 선택 메모로 초기 맥락을 보완할 수 있다.
  - 상세 정보는 관계 형성 후 받는 편이 완료율이 높고 매칭 판단 정확도가 높다.
- 대안:
  - 대안 A: 기존 단일 폼 유지 + 필드 축소
  - 기각 이유: 운영 단계 구분이 사라져 후속 액션 우선순위 관리가 어려움

### 2) 메시지 정책: 고정 기간 제거, 단계/상태 기반 제안
- 결정:
  - Hero/CTA에서 "4주" 같은 고정 기간 약속을 제거한다.
  - 사용자 단계에 맞춘 전략 지원 문구를 채택한다.
- 이유:
  - 타겟의 준비도 편차가 커 고정 기간 약속은 신뢰를 떨어뜨릴 수 있다.
- 대안:
  - 대안 A: 4~8주 범위 표기
  - 기각 이유: 여전히 시간 중심 기대치를 과도하게 고정함

### 3) 데이터 모델: 운영 상태값 명시
- 결정:
  - 리드 상태를 `new -> strategy_in_progress -> matching_pending -> intro_in_progress -> closed`로 정의한다.
  - 이벤트 추적은 `foreign_quick_intake_submit`, `foreign_matching_profile_complete`, `foreign_intro_started`를 추가한다.
- 이유:
  - 단순 문의 수 대신 실제 운영 단계 성과를 측정할 수 있다.
- 대안:
  - 대안 A: 기존 `contact_submit` 하나만 유지
  - 기각 이유: 어떤 단계에서 병목이 생기는지 분석 불가

상태값 정의:
- `new`: 1차 quick intake 제출 직후
- `strategy_in_progress`: 전략 정리/초기 상담 진행 중
- `matching_pending`: 소개 가능성 검토를 기다리는 상태(소개 시작 전)
- `intro_in_progress`: 소개/연결 프로세스 진행 중
- `closed`: 목표 달성 또는 중단으로 종료

### 4) 범위 경계 문구 위치 고정
- 결정:
  - 범위 경계(취업 보장 아님, 비자/법률 직접 대행 아님)를 Hero 하단 보조 문구 + FAQ + 폼 근처에 반복 노출한다.
  - 광주/전남 대면 우선, 타지역 온라인 우선 원칙을 Hero 보조 문구 + FAQ에 함께 노출한다.
- 이유:
  - 사용자 오해를 줄이고 범위 외 문의 비용을 줄인다.
  - 지역 지원 방식에 대한 기대치를 사전에 정렬할 수 있다.
- 대안:
  - 대안 A: FAQ에만 배치
  - 기각 이유: CTA 직전 기대치 정렬이 되지 않음

### 5) 커뮤니티 섹션은 "정보 공유/대기열" 수준으로 시작
- 결정:
  - 커뮤니티는 초기에는 별도 채널을 즉시 개설하지 않고 대기열/안내 중심으로 운영한다.
  - 커뮤니티 채널은 운영 임계치 도달 시점에 개설한다.
  - 임계치는 다음 세 조건으로 정의한다:
    - 최근 4주 활성 후보 30명 이상
    - 주간 반복 질문 10건 이상
    - 운영자 모더레이션 주 2회 이상 가능
- 이유:
  - 초기 운영 부담을 줄이면서도, 채널 개설 시점의 품질과 지속성을 확보할 수 있다.
- 대안:
  - 대안 A: 즉시 Discord/Open chat 개설
  - 기각 이유: 초기 인원 부족 시 채널이 비활성 상태가 되어 신뢰를 저하시킬 수 있음
  - 대안 B: 사이트 내 커뮤니티 기능 직접 구현
  - 기각 이유: 범위 과대, 운영 복잡도 급증

### 5-1) 커뮤니티 대기열 측정 이벤트
- 결정:
  - `foreign_community_waitlist_submit` 이벤트를 추가해 대기열 성장률을 추적한다.
- 이유:
  - 채널 개설 임계치 판단 근거를 정량적으로 확보할 수 있다.

### 6) Open Questions 해소 결정
- 결정:
  - 2차 프로필 수집은 별도 URL로 분리한다.
  - `matching_pending` 진입 기준은 초기에는 수동 판단 + 체크리스트 기반으로 운영한다.
- 이유:
  - 별도 URL은 재방문/저장/트래킹/운영 관리가 단순하며, 수동 기준은 초기 데이터가 적은 구간에서 오판을 줄인다.
- 체크리스트(초기안):
  - `cv_or_linkedin` 제출
  - `github_or_portfolio` 제출
  - `tech_stack`과 `experience_level` 기재
  - `visa_status`, `work_preference`, `location_preference`, `available_from` 기재 완료

### 7) Hero CTA hierarchy by persona
- 결정:
  - 외국인 페이지는 `Quick Intake`를 첫 행동으로 유지하고 나머지 행동은 후속 단계로 배치한다.
  - 메인 창업가 페이지는 `상담 문의하기`를 주 CTA로 유지하고, 서비스 보기는 링크 수준으로 하향 조정한다.
- 이유:
  - 두 페르소나 모두 첫 행동이 하나로 보일 때 전환 판단이 단순해진다.

## Risks / Trade-offs

- [Risk] 1차 CTA에서 정보가 너무 적어 후속 응답 품질이 낮을 수 있음
  - Mitigation: 1차 제출 즉시 2차 입력 유도 메일/메시지 템플릿 제공
- [Risk] 2차 전환율이 낮을 수 있음
  - Mitigation: 전략 콜 이후 "필수 5항목" 버전의 간이 2차 폼 제공
- [Risk] 소개 가능성 문구가 취업 보장으로 오해될 수 있음
  - Mitigation: 모든 CTA 주변에 비보장 문구 반복 배치
- [Risk] 운영자가 상태값을 수동 갱신하지 않으면 KPI 신뢰도 저하
  - Mitigation: 관리자 화면에서 기본 상태/전이 규칙과 점검 루틴 문서화

## Migration Plan

1. 콘텐츠/카피 변경: Hero, 서비스, FAQ, 경계 문구를 단계형 퍼널 기준으로 개편
2. 폼/뷰 변경: 1차/2차 CTA 엔드포인트 및 필드 분리
3. 데이터/이벤트 변경: 상태값 필드(또는 메타데이터)와 신규 퍼널 이벤트 추가
4. 운영 전환: 기존 문의는 `legacy` 또는 `new`로 매핑해 초기 상태 정리
5. 롤백 전략: 단일 문의 폼 템플릿과 기존 이벤트만 사용하는 버전으로 즉시 복귀 가능하도록 유지
