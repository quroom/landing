## Context

서비스 카피는 전환을 위해 읽기 쉬워야 하고, 결과물 문구는 분쟁 리스크를 줄이기 위해 책임 범위가 명확해야 한다.

현재 문제는 두 축으로 나뉜다.
1. 톤 문제: 카드 문장이 모두 비슷한 관리형 표현으로 읽혀 차별점이 약함
2. 범위 문제: 일부 결과물 표현이 보장 범위를 넘어서 해석될 여지 존재

## Goals / Non-Goals

**Goals**
- 서비스 카드 문장을 실행형 톤으로 단순화하고 읽기 리듬을 개선한다.
- 카드 결과물 문구를 "실제 제공/인계 가능한 산출물" 중심으로 제한한다.
- 외주용역 트랙의 후속 지원은 범위/기간 별도 합의 원칙을 명시한다.

**Non-Goals**
- 서비스 가격/기간 자체 변경
- 계약서 조항 자동 생성
- 운영 지원 SLA 도입

## Decisions

### 1) 결과물 문구는 책임 가능한 산출물로 한정
- 카드1: `진단 요약 문서 + 2주 실행 후보 리스트`
- 카드2: `운영 가능한 자동화 구성 + 자동화 운영 가이드 문서`
- 카드3: `구축 결과물 + 운영 이관 문서 (후속 지원 범위 별도 합의)`

### 2) 카드3 운영 이관 문서 최소 범위를 고정
운영 이관 문서는 다음 항목을 포함한다.
- 구성 개요: 개발 스택, PaaS/인프라, 연동 구조
- 배포/실행: 기본 배포 절차, 서버 재시작 방법, 롤백 기본 절차
- 환경설정: 필수 환경변수 목록(값 제외), 비밀정보 관리 위치
- 운영 점검: 로그 위치, 1차 장애 점검 순서, 빈발 이슈 대응
- 지원 채널: 플랫폼 고객센터/지원 경로, 내부 에스컬레이션 경로

### 3) 문체는 실행형으로 통일
- 원칙: 명사형 나열보다 "무엇을, 언제까지, 어떤 상태로" 문장 사용
- 금지: 기간/범위 미정 지원을 확정 약속처럼 표현

### 4) 운영 이관 문서 템플릿은 관리자 전용 페이지에서 조회 가능하게 한다
- 결정: 운영 이관 문서 템플릿을 관리자 전용 문서 페이지(운영 가이드 영역)에서 조회 가능하게 한다.
- 이유: 상담/제안/인수인계 단계에서 동일 기준 문구를 참조해 기대치 불일치를 줄일 수 있다.
- 범위: 템플릿 조회 기능만 포함하며, 고객 노출 페이지에는 직접 노출하지 않는다.
- 정보 구조: `단일 템플릿형 + 카드별 앵커 목차`를 기본 구조로 채택한다.

### 5) 상담 자동응답 메일에는 결과물 범위 요약을 포함하지 않는다
- 결정: 상담 자동응답 메일 본문에는 결과물 범위 요약을 넣지 않는다.
- 이유: 메일 본문 과밀화를 방지하고, 범위 문구는 상담/제안 단계에서 맥락 기반으로 안내한다.

## Risks / Trade-offs

- [Risk] 문구가 지나치게 보수적이면 세일즈 임팩트가 약해질 수 있음
  - Mitigation: 요약 문장은 실행형으로 유지하고, 결과물 라인만 보수적으로 고정
- [Risk] 운영 이관 문서 범위가 프로젝트 특성과 불일치할 수 있음
  - Mitigation: "기본 포함 항목" + "프로젝트별 조정 가능" 문구 병행
