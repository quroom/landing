## Context

현재 리드마그넷 플로우는 진단 제출과 이메일 발송까지는 동작하지만, 운영 관점에서 병목을 빠르게 파악할 수 있는 전환 측정 기준이 부족하다. 특히 이벤트는 기록되고 있으나 KPI 공식, 대시보드 노출 기준, GA4/내부 데이터 역할 분리가 명확히 문서화되어 있지 않아 개선 반복 속도가 느려진다.

이번 변경은 신규 외부 인프라를 도입하지 않고, 기존 `FunnelEvent` 기반 데이터를 활용해 즉시 확인 가능한 MVP 퍼널 대시보드를 제공하고, 이후 확장 가능한 측정 계약을 고정하는 데 초점을 둔다.

## Goals / Non-Goals

**Goals:**
- 리드마그넷 퍼널 KPI(제출율/메일성공률/상담전환율) 계산 기준을 고정한다.
- 이벤트 키/메타 스키마를 운영 문서와 스펙에 일치시키고 누락 기준을 정의한다.
- 관리자 대시보드에 간단 확인용 퍼널 카드(카운트+비율)를 추가한다.
- 분모 0 처리 등 운영 안정성 규칙을 명시한다.
- 이후 실험(카피/CTA) 확장을 위한 최소 데이터 계약을 확보한다.

**Non-Goals:**
- GA4 Admin API 자동화 스크립트 구현은 이번 범위에서 제외한다.
- 고급 코호트 분석/멀티터치 어트리뷰션은 이번 범위에서 제외한다.
- 신규 BI 도구 도입/데이터 웨어하우스 구축은 이번 범위에서 제외한다.

## Decisions

### 1) 내부 대시보드를 1차 소스, GA4를 채널 보조 소스로 분리한다
- 선택: 운영 KPI 판정은 내부 이벤트 집계(FunnelEvent) 기준으로 고정하고, GA4는 유입/캠페인 보조 분석에 사용한다.
- 이유: 서버 이벤트는 리드마그넷 문맥(`lead_source`, `lead_context`, `grade`) 보존에 유리하고, 수치 불일치 시 원인 분석이 빠르다.
- 대안: GA4 수치만 단일 기준으로 사용.
  - 미채택 이유: 이벤트 샘플링/설정 변경 영향으로 운영 의사결정 기준이 흔들릴 수 있다.

### 2) KPI는 3개 Primary + 2개 Secondary로 고정한다
- 선택: `submit_rate`, `mail_success_rate`, `contact_conversion_rate`를 1차 KPI로, `home_to_start_rate`, `grade_distribution`을 보조 KPI로 고정한다.
- 이유: 퍼널 병목을 빠르게 식별하고 실험 우선순위를 정하기에 최소/충분 집합이다.
- 대안: 초기부터 지표 다수 확장.
  - 미채택 이유: 운영 해석 복잡도만 늘고 실행 속도가 저하된다.

### 3) 대시보드는 MVP 카드형으로 시작한다
- 선택: 관리자 페이지에 `start / submit / mail_sent / contact_submit` 카운트와 3개 비율을 카드로 노출한다.
- 이유: 구현 비용 대비 확인 속도가 가장 빠르고, 주간 운영 리뷰에 즉시 사용 가능하다.
- 대안: 차트 중심 고급 대시보드부터 구축.
  - 미채택 이유: 초기 개발량이 커져 개선 루프 시작 시점이 늦어진다.

### 4) 이벤트 메타 표준 키를 고정한다
- 선택: 공통 키(`event_name`, `page_key`, `lead_source`)와 문맥 키(`score`, `grade`, `inquiry_type`, `lead_context`, `utm_*`)를 표준으로 유지한다.
- 이유: 향후 GA4/내부 비교 시 스키마 정합성 문제를 줄인다.
- 대안: 이벤트별 임의 키 허용.
  - 미채택 이유: 지표 정의/비교 쿼리 재사용성이 낮아진다.

### 5) 메일 성공 이벤트를 사용자용/관리자용으로 분리한다
- 선택: `lead_magnet_email_sent_user`(KPI용)와 `lead_magnet_email_sent_admin`(운영 알림용)을 분리한다.
- 이유: 관리자 알림 메일까지 KPI 분자에 포함되면 성공률이 100%를 초과할 수 있어 지표 신뢰도가 떨어진다.
- 대안: 기존 `lead_magnet_email_sent` 단일 이벤트 유지.
  - 미채택 이유: 전환 지표와 운영 알림 지표를 분리할 수 없어 해석 오류가 반복된다.

### 6) 분모 0 규칙은 0.0%로 강제한다
- 선택: 모든 비율 계산에서 분모 0일 때 `0.0`으로 표시한다.
- 이유: 운영 화면에서 NaN/예외 노출을 방지하고 해석 일관성을 유지한다.
- 대안: 빈값 또는 "N/A" 표기.
  - 미채택 이유: 필터/기간 변경 시 비교 자동화가 어려워진다.

## Risks / Trade-offs

- [내부 이벤트 누락으로 KPI 왜곡] -> 이벤트 발생 지점별 회귀 테스트와 운영 점검 체크리스트를 추가한다.
- [GA4와 내부 수치 불일치 혼선] -> 운영 기준은 내부 KPI로 고정하고, GA4는 채널 최적화 용도로 명시한다.
- [관리자 알림 이벤트 혼입으로 메일 성공률 왜곡] -> 사용자용/관리자용 메일 이벤트를 분리하고 KPI 분자에는 사용자용만 사용한다.
- [개인정보 이벤트 전송 리스크] -> 이벤트 페이로드에서 PII를 금지하고 비식별 메타만 허용한다.
- [MVP 카드형 한계로 인사이트 부족] -> 주간 운영 2회 후 필요 차트만 단계적으로 확장한다.
- [이벤트 키 변경 시 리포트 단절] -> 이벤트 키 변경은 별도 change로 관리하고 백필/마이그레이션 규칙을 문서화한다.

## Migration Plan

1. 이벤트 집계 키와 KPI 수식을 코드/문서에서 동일하게 고정한다.
2. 관리자 대시보드에 리드마그넷 퍼널 카운트+비율 카드를 추가한다.
3. 기본 테스트에 퍼널 카드 노출과 분모 0 처리 검증을 추가한다.
4. `codex-document/ga4-setup-manual.md`에 측정 기준/운영 규칙을 반영한다.
5. 주간 리뷰에서 병목 단계를 확인하고 다음 실험 change를 정의한다.

Rollback:
- 대시보드 카드 노출만 비활성화하고 기존 이벤트 카운트 뷰로 즉시 복귀 가능하게 유지한다.
