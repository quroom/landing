## Context

리드마그넷 결과는 현재 웹 Preview와 이메일 발송 경로가 기능적으로 연결되어 있지만, 출력 텍스트 조합 지점이 완전히 동일하지 않아 문구/섹션/링크 규칙이 분기될 위험이 있다.  
최근 변경에서 "최약 카테고리 1개 기본 노출", "CTA 1개", "`#contact` 정규화" 같은 정책이 강화되었고, 이후 카피 수정이 반복될 가능성이 높아 단일 렌더 계약이 필요하다.

## Goals / Non-Goals

**Goals:**
- Preview와 이메일이 동일한 섹션 데이터 계약을 사용하도록 구조를 정리한다.
- 섹션 순서(요약 -> 최약 카테고리 -> 2주 실행 -> CTA)와 CTA 링크 정규화 결과가 채널 간 동일하게 유지되도록 한다.
- 텍스트/HTML 렌더 출력이 같은 입력 모델에서 생성되도록 하여 회귀 테스트 지점을 단순화한다.

**Non-Goals:**
- 진단 점수 계산 로직 및 카테고리 산출 로직 변경
- 기본 노출 정책(최약 카테고리 1개) 변경
- 메일 제목/등급 체계(A/B/C) 자체 변경

## Decisions

### 1) 공통 섹션 모델을 먼저 만들고 채널 렌더를 분리한다
- 선택: `landing` 레이어에 `build_lead_magnet_sections(...)` 성격의 공통 섹션 빌더를 두고, Preview/이메일은 이 결과를 소비한다.
- 이유: 채널별 마크업은 달라도 "무엇을 보여줄지" 규칙을 1곳에서 유지할 수 있다.
- 대안: Preview와 이메일 각각에서 동일 규칙을 병렬 유지.
  - 미채택 이유: 카피/순서 변경 시 동시 수정 누락 가능성이 높다.

### 2) CTA 링크 정규화는 공통 유틸 단일 경로로 강제한다
- 선택: `#contact`, `/#contact`, 절대 URL 입력을 공통 정규화 함수에서 처리해 두 채널이 동일 값을 사용한다.
- 이유: 메일 클라이언트/브라우저 컨텍스트 차이로 링크 오동작이 발생할 수 있어, 입력 허용은 유연하게 두되 출력은 표준화해야 한다.
- 대안: 각 채널에서 개별 정규화.
  - 미채택 이유: URL 형식 회귀가 다시 발생한다.

### 3) "동등성 테스트"를 1차 회귀 방어선으로 둔다
- 선택: 섹션 단위 비교(제목/순서/CTA 링크)와 텍스트 스냅샷 검증을 병행한다.
- 이유: 구조 계약 위반과 카피/줄바꿈 회귀를 동시에 잡을 수 있다.
- 대안: 섹션 단위 비교만 수행.
  - 미채택 이유: 실제 사용자 노출 문구 회귀를 놓칠 수 있다.

### 4) 공통 빌더 출력은 채널 중립 AST로 유지한다
- 선택: 공통 빌더는 문자열이 아닌 채널 중립 AST(섹션 객체 목록)를 반환하고, 채널별 렌더러가 이를 텍스트/HTML로 변환한다.
- 이유: 채널 간 계약을 명확히 유지하고, 테스트에서 구조 비교를 안정적으로 수행할 수 있다.
- 대안: 공통 빌더가 텍스트/HTML 문자열까지 직접 조합.
  - 미채택 이유: 채널별 표현 차이를 흡수하기 어렵고 변경 시 회귀 추적이 불투명해진다.

### 5) 기존 함수 시그니처는 최대한 유지하고 내부만 치환한다
- 선택: 외부 호출부(`views`, `mailers`)의 인터페이스를 유지한 채 내부 렌더링 데이터를 공통 빌더로 전환한다.
- 이유: 영향 범위를 최소화해 점진적으로 반영하고, 롤백 시 위험을 줄인다.
- 대안: 새 서비스 객체 도입 후 호출부 대규모 교체.
  - 미채택 이유: 이번 변경 범위 대비 과도하며 테스트 보정 비용이 크다.

## Risks / Trade-offs

- [공통화 과정에서 기존 텍스트/HTML 포맷 미세 차이 손실] -> 가독성 관련 핵심 규칙(line-height, 문단 간격, 섹션 구분)을 테스트 기대값으로 고정한다.
- [공통 모델이 커져 향후 확장 난이도 상승] -> 섹션 모델을 "핵심 필드 + 선택 필드"로 분리해 단순 구조를 유지한다.
- [Preview 요구와 이메일 요구가 장기적으로 달라질 가능성] -> 공통 영역과 채널 특화 영역의 경계를 명시하고, 특화 필드는 후처리 단계에서만 추가한다.

## Migration Plan

1. 현재 Preview/이메일 생성 경로에서 공통 요소(섹션 순서, CTA, 카테고리 기본 노출)를 추출해 공통 빌더 함수로 이동한다.
2. `landing/views.py` Preview 경로를 공통 빌더 기반으로 전환한다.
3. `landing/mailers.py` 텍스트/HTML 생성 경로를 같은 빌더 입력으로 전환한다.
4. `landing/tests/test_contact_form.py`, `landing/tests/test_landing_pages.py`에 동등성/정규화 검증을 추가한다.
5. 회귀 확인 후 적용하고, 문제 시 기존 렌더 함수 경로로 즉시 롤백한다.
