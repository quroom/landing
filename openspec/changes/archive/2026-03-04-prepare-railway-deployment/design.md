## Context

현재 서비스는 로컬 실행 기준으로 운영되고 있으며, Railway 배포에 필요한 런타임 계약(환경변수, 시작 커맨드, 정적 파일, 마이그레이션 실행 타이밍)이 문서와 코드에서 명확히 고정되어 있지 않다. 이 상태에서는 배포 시점마다 수동 확인 항목이 달라지고, `DEBUG=False` 운영 모드에서만 발생하는 장애를 사전에 차단하기 어렵다.

이번 변경은 배포 성공 자체보다 "반복 가능한 배포 절차"를 만드는 데 초점을 둔다. 특히 기존 운영 지표(문의/메일/대시보드) 흐름이 Railway 전환 후에도 동일하게 동작하도록 안전 가드를 정의한다.

## Goals / Non-Goals

**Goals:**
- Railway 운영 배포를 위한 실행 계약(필수 환경변수, 런 커맨드, 정적 파일, DB 연결)을 명시한다.
- 운영 모드(`DEBUG=False`)에서 누락되면 장애가 나는 설정을 사전 검증 가능하도록 정리한다.
- 배포 직후 점검 루틴(헬스체크, 관리자 접근, 메일 경로, 핵심 폼 제출)을 체크리스트로 표준화한다.
- 배포 실패 시 복구 가능한 최소 롤백 절차를 정의한다.

**Non-Goals:**
- 인프라 벤더 다중화(AWS/GCP 동시 지원) 설계.
- 애플리케이션 기능 확장(신규 화면/비즈니스 로직 추가).
- 관측 스택 전면 개편(로그 수집/모니터링 플랫폼 교체).

## Decisions

### 1) Railway 배포 기준을 "코드+문서" 동시 계약으로 관리
- 선택: 런타임 필수값을 settings 및 배포 문서에 동시에 명시한다.
- 이유: 문서만 있으면 드리프트가 발생하고, 코드만 있으면 운영자가 맥락을 잃는다.
- 대안: Railway UI 설정만 신뢰.
  - 미채택 이유: 팀 변경/재배포 시 누락 위험이 크다.

### 2) 운영 모드 안전성 우선: 필수 환경변수 누락 시 빠르게 실패
- 선택: `DEBUG=False`에서 핵심 값(`SECRET_KEY`, DB 연결, `ALLOWED_HOSTS`) 검증을 강화한다.
- 이유: 부분 동작 상태로 올라오면 장애 탐지가 늦어지고 데이터 일관성 문제가 커진다.
- 대안: 기본값으로 완화하고 런타임 경고만 노출.
  - 미채택 이유: 잘못된 기본값이 실제 트래픽으로 이어질 수 있다.

### 3) 마이그레이션은 배포 파이프라인 단계로 고정
- 선택: 릴리즈 시점에 명시적으로 마이그레이션을 실행하고, 완료 후 앱 트래픽을 받는다.
- 이유: 스키마 불일치 상태에서 앱이 먼저 뜨는 문제를 방지한다.
- 대안: 앱 시작 시 자동 마이그레이션.
  - 미채택 이유: 재시작/스케일링 시 예측 불가능한 실행 타이밍이 생긴다.

### 4) 정적 파일은 배포 시점에 수집 완료된 상태를 기준으로 운영
- 선택: `collectstatic`을 배포 절차에 포함하고 결과를 검증한다.
- 이유: 운영 모드에서 CSS/JS 누락은 즉시 사용자 경험 장애로 이어진다.
- 대안: 개발 모드와 동일하게 앱이 정적 파일을 즉시 서빙.
  - 미채택 이유: 운영 성능/안정성 측면에서 비권장.

### 5) 배포 후 운영 확인은 관리자 대시보드와 핵심 사용자 경로를 함께 점검
- 선택: `/admin-dashboard/` 접근, 문의 제출, 리드마그넷 제출/메일 발송 흐름을 최소 확인 세트로 고정.
- 이유: 단순 헬스체크만으로는 실제 전환 퍼널 장애를 발견하기 어렵다.
- 대안: 헬스체크 200만 확인.
  - 미채택 이유: 비기능적 성공만 확인하고 비즈니스 실패를 놓친다.

### 6) 헬스체크 엔드포인트는 별도 `/healthz`로 분리
- 선택: 서비스/의존성 상태 확인용 경량 엔드포인트 `/healthz`를 별도로 둔다.
- 이유: 일반 페이지 라우팅/템플릿/캐시 영향 없이 배포 전후 상태를 일관되게 확인할 수 있다.
- 대안: 기존 `/` 또는 특정 랜딩 URL을 헬스체크로 재사용.
  - 미채택 이유: 콘텐츠 변경/템플릿 오류가 인프라 헬스체크와 섞여 원인 분석이 어려워진다.

### 7) 운영 환경 검증은 "앱 시작 강제 + 관리 명령 사전 점검"을 병행
- 선택: `DEBUG=False`에서 앱 시작 시 필수 환경변수 검증을 강제하고, 별도 관리 명령으로 배포 전 사전 점검도 제공한다.
- 이유: 시작 차단으로 치명적 오설정을 막고, 사전 점검으로 배포 파이프라인 실패를 앞단에서 줄일 수 있다.
- 대안: 시작 강제만 적용 또는 사전 점검만 적용.
  - 미채택 이유: 단일 방식은 운영 안정성과 배포 편의성 중 한쪽이 약해진다.

### 8) 배포 직후 점검은 CI/CD 단계까지 확장
- 선택: 수동 체크리스트를 유지하되, 동일 항목을 CI/CD post-deploy 단계에서 자동 검증한다.
- 이유: 사람이 확인하던 절차를 자동화해 배포 품질 편차를 줄일 수 있다.
- 대안: 수동 체크만 유지.
  - 미채택 이유: 반복 배포에서 누락 위험이 누적된다.

### 9) 관리자 운영 검증 링크를 단일 허브 페이지로 제공
- 선택: 배포 점검에 필요한 URL(`/healthz`, `/admin-dashboard/`, 핵심 랜딩/폼)을 한 페이지에서 바로 열 수 있는 관리자용 링크 허브를 제공한다.
- 이유: 운영자가 URL을 개별 기억하지 않아도 점검 순서를 빠르게 수행할 수 있다.
- 대안: 기존 문서 체크리스트만 유지.
  - 미채택 이유: 실제 운영 시 문서와 브라우저 이동이 분리되어 점검 누락이 발생하기 쉽다.

## Risks / Trade-offs

- [Risk] 환경변수 검증 강화로 초기 배포 실패 빈도 증가 → Mitigation: 필수값 체크리스트와 `.env.sample` 동기화.
- [Risk] 마이그레이션 단계 추가로 배포 시간 증가 → Mitigation: 사전 리허설 및 변경량 작은 단위 배포 유지.
- [Risk] Railway 플랫폼 설정과 코드 설정 간 불일치 → Mitigation: 문서 기준 단일화 및 배포 전 self-check 스크립트화.
- [Risk] SMTP/외부 연동 장애 시 배포 성공처럼 보일 수 있음 → Mitigation: 배포 후 테스트 메일 발송과 이벤트/상태 확인을 필수화.

## Migration Plan

1. 배포 준비
- Railway 서비스에 필수 환경변수 등록 (`SECRET_KEY`, DB, 메일, `ALLOWED_HOSTS`, `CSRF_TRUSTED_ORIGINS` 등).
- 릴리즈 커맨드에 `migrate` 및 `collectstatic` 포함 여부 확인.

2. 사전 검증
- 로컬에서 운영 모드 유사 설정으로 `check`, `test` 실행.
- 마이그레이션 체인/스쿼시 파일 영향 확인.

3. 배포 실행
- Railway 배포 트리거 후 빌드/릴리즈 로그 확인.
- 마이그레이션 완료 및 앱 부팅 성공 확인.

4. 배포 직후 점검
- `/healthz` 200 응답 확인(필수).
- 운영 링크 허브 페이지를 통해 홈페이지/핵심 랜딩 접근 확인.
- 운영 링크 허브 페이지를 통해 관리자 대시보드 접근 및 기본 카드 노출 확인.
- 문의 폼/리드마그넷 제출 테스트, 메일 발송/상태 기록 확인.

5. 자동화 실행 로직
- Pre-deploy: 관리 명령으로 환경변수/DB/SMTP/정적파일 설정 점검(실패 시 배포 중단).
- Deploy: `migrate`, `collectstatic`, 앱 기동 순서 고정.
- Post-deploy: CI/CD에서 `/healthz`, `/admin-dashboard/` 접근, 핵심 폼 submit smoke test 자동 실행.

6. 롤백 전략
- 장애 발생 시 직전 안정 릴리즈로 롤백.
- 롤백 후 DB 스키마 호환성 확인(비가역 변경 여부 사전 점검 필수).
