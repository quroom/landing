## Context

현재 배포 점검 체계는 `check_deploy_ready`(사전 검증), `/healthz/`(기본 상태), `post-deploy-smoke.sh`(사후 검증), 운영 링크 페이지(수동 점검 진입점)로 구성되어 있습니다. 다만 `/healthz/`가 단순 프로세스 생존 확인 수준이고, smoke check가 일부 경로에 대해 `404` 여부만 확인해 실패 원인 구분이 어렵습니다. 또한 운영자는 링크를 열어가며 직접 판단해야 해서 배포 직후 점검 누락 가능성이 있습니다.

이 change는 Railway 배포 신뢰도(자동 검증 정확성)와 운영 가시성(관리자 점검 속도)을 동시에 강화하는 것을 목표로 합니다.

## Goals / Non-Goals

**Goals:**
- 배포 상태 신호를 `liveness/readiness`로 분리해 헬스체크 의미를 명확히 한다.
- post-deploy smoke 검증을 경로별 기대 상태코드 기반으로 강화한다.
- 운영 링크 페이지를 "링크 모음"에서 "점검 결과 포함 허브"로 확장해 운영자 판단 시간을 줄인다.
- 배포 실패 시 재시도/롤백 기준을 runbook과 CI 로그에서 일관되게 추적 가능하게 만든다.

**Non-Goals:**
- Railway 배포 자체를 완전 무중단/blue-green으로 전환하지 않는다.
- 외부 모니터링 SaaS(예: Datadog, NewRelic) 연동은 이번 범위에서 제외한다.
- 고객용 페이지 UX를 변경하지 않는다(관리자 운영 흐름 중심).

## Decisions

- 선택: 헬스체크를 `liveness`와 `readiness`로 분리하고, `/healthz/`는 통합 요약 엔드포인트로 유지한다.
- 이유: 기존 경로 호환성을 유지하면서도 배포 파이프라인에서 신호 해석을 명확히 할 수 있다.
- 대안: `/healthz/` 하나만 유지하고 내부 필드만 추가.
  - 미채택 이유: 배포 단계별 실패 원인을 빠르게 분기하기 어렵다.

- 선택: smoke check를 "404 아님" 기준에서 "경로별 기대 상태코드" 기준으로 전환한다.
- 이유: 접근제어/HTTP 메서드 정책 문제를 조기에 탐지할 수 있다.
- 대안: 기존 스크립트 유지 + 수동 확인 강화.
  - 미채택 이유: 수동 확인 의존도가 높아 배포 누락 위험이 남는다.

- 선택: 운영 링크 페이지에 최근 점검 요약(최근 실행 시각, 결과, 실패 항목)을 표시한다.
- 이유: 관리자가 링크를 모두 열지 않아도 배포 직후 상태를 한 번에 파악할 수 있다.
- 대안: 별도 관리자 대시보드 신설.
  - 미채택 이유: 범위 대비 구현량이 커지고 기존 운영 동선과 분리된다.

- 선택: CI workflow는 pre-deploy 필수, post-deploy는 `base_url` 제공 시 자동 수행 구조를 유지하되 결과 해석 규칙을 명시한다.
- 이유: 현재 운영 방식과 호환되며 단계적으로 자동화를 확장할 수 있다.
- 대안: post-deploy를 항상 강제 수행.
  - 미채택 이유: 임시/사설 환경에서 URL 미제공 실행이 막힐 수 있다.

## Risks / Trade-offs

- [헬스체크 경로 추가로 운영 혼선] → runbook에 경로별 의미(`/healthz`, `/healthz/live`, `/healthz/ready`)를 명시하고 운영 링크 페이지에 설명 노출
- [엄격한 상태코드 검증으로 초기 false failure 증가] → 첫 단계에서는 기대 코드 테이블을 최소 경로부터 도입하고 점진 확장
- [운영 링크 페이지 책임 증가] → 상세 장애 분석은 기존 로그/CI 페이지에서 수행하고 운영 링크 페이지는 "요약 + 진입점" 역할로 제한

## Migration Plan

1. 헬스체크 엔드포인트 분리 추가 및 기존 `/healthz/` 호환 유지
2. smoke script를 상태코드 기반 검증으로 전환하고 runbook 기대코드 표 업데이트
3. 운영 링크 페이지에 최근 점검 요약 카드 추가
4. CI(`deploy-readiness.yml`)에서 post-deploy 결과 표시 포맷 정리
5. 배포 후 운영 점검: `/healthz*` 응답, 운영 링크 페이지 요약, smoke 통과 여부 확인

Rollback:
- 새 엔드포인트/요약 카드 문제 발생 시 기존 `/healthz/` 기반 경량 점검으로 즉시 회귀
- smoke strict 모드가 과도한 실패를 일으키면 임시로 완화 모드(핵심 경로만)로 되돌린다

## Open Questions

- readiness에서 데이터베이스 외에 SMTP 연결 검증까지 포함할지, 배포 성능을 위해 설정 검증만 유지할지?
- 운영 링크 페이지의 "최근 점검 결과" 저장 위치를 DB로 둘지, CI 산출물 링크 기반으로 둘지?
- post-deploy 자동화 실패 시 자동 롤백 트리거까지 이번 범위에 포함할지, 알림/수동 롤백으로 제한할지?
